# Cartel — Full-Loop Build Plan (2026-07-18)

Close the demo loop: **dashboard → cart → split → rule → approve (realtime) →
checkout (charge cards) → savings + dashboard update.**

**Current state:** all 8 API routes exist and pass. Cart→split works on real
data. Approval device is wired. What remains is **frontend wiring + 3 small
route additions + 2 integration-seam fixes.** No new architecture.

Everything below is relative to `web/`. Integer cents everywhere; format at
render. Build/verify from repo root or `web/` (`npm run …` delegates).

Legend: **fix** = correct an existing seam · **build** = new code · gates
`I1–I4` are the master plan's integration gates.

## Task A — Real user identity  (unblocks everything)

**Problem:** `useMe`/`UserSwitcher`/`SplitView`/dashboard resolve people from the
mock `lib/data.people` (ids `ava/sam/…`). The seeded users are `Sam/Priya/Alex/
Jordan` with **UUIDs**, so `?user=sam` never matches a real approver and per-user
splits can't be labeled.

- [ ] **build** `GET /api/users` → `app/api/users/route.ts`
  - `admin()`, resolve the single household (same pattern as dashboard), return
    `{ users: [{ id, name, color }] }` ordered by name.
- [ ] **build** `lib/useUsers.ts` — client hook: `fetch('/api/users')`, cache in
  module scope, return `{ users, byId }`. Falls back to `lib/data.people` on error
  (offline).
- [ ] **fix** `lib/useMe.ts` — seed `me` from `/api/users` (first user's real id),
  keep `?user=` override + localStorage. `users` now comes from `useUsers`.
- [ ] **fix** `components/UserSwitcher.tsx` — roster from `useUsers` (real ids);
  avatars use `user.color`.
- [ ] **fix** `components/SplitView.tsx` — map `userId → {name,color}` via
  `useUsers().byId` instead of `lib/data.people` (unknown id still degrades
  gracefully).
- [ ] **fix** cart flow — `app/cart/page.tsx` `build()` sends `{ text, userId: me }`
  so `purchases.created_by` is a real user (`useMe()` in the page).
- **Verify:** switch user in the header → `/approve?user=<uuid>` link uses the real
  id; `GET /api/approvals?user=<uuid>` returns that user's pending rows.

## Task B — Checkout flow + savings money-shot  (I3 tail, I4)

**Problem:** after split there is no "check out" action; `SavingsCard` is never
rendered; and `summary` reads savings from `purchase.note` as JSON but
`cart/build` stores the raw prompt string there → `savedCents` is always 0.

- [ ] **fix seam** `app/api/cart/build/route.ts` — persist `note` as
  `JSON.stringify({ text, skipped })` (currently `note: text`). `insertPurchaseItems`
  is unaffected; `summary`'s `parseSkippedItems` then finds the skipped list.
  - Keep the API response shape identical (still returns `{purchaseId, items, skipped}`).
  - `agent-data.createBuildingPurchase({ note })` already stores whatever string it's
    handed — pass the JSON from the route.
- [ ] **build** checkout phase in `app/cart/page.tsx` (phases: `input → cart →
  split → awaiting? → checkout → done`).
  - After `split`, if any line has `flag` → purchase is `awaiting_approval`.
    Show an **"Awaiting approval"** state that polls `GET /api/approvals?user=…`
    (or re-POSTs split to read status) until no pending remain, then enables
    checkout. No flags → checkout enabled immediately.
  - **"Check out"** → `POST /api/purchase/[id]/checkout` → `{ charges:
    [{userId, amountCents, status}] }`. Handle 409 (still pending) by staying in
    the awaiting state.
- [ ] **build** `components/ChargeCards.tsx` — one card per charge, animating to
  "charged" with staggered `CheckBadge` (reuse the `SavingsCard`/dashboard card
  styling). This is the money-shot.
- [ ] **wire** `components/SavingsCard.tsx` (already built) — after charges land,
  `GET /api/purchase/[id]/summary` → render with real `savedCents`/`stepsCollapsed`.
- **Verify:**
  - No-flag cart → checkout → 4 `succeeded` charges → SavingsCard shows non-zero
    savings (paper-towels skip).
  - Flagged cart → checkout blocked (409) → approve on device → checkout unblocks
    → charges animate. (**I3 tail + I4**)

## Cross-cutting: thread "who am I"

`useMe()` is the single source of the current user. It flows to: cart build
(`createdBy`), settings (`userId`), and the header approve link. `?user=` on the
approval device overrides it and is never persisted. After Task A these are all
real UUIDs.

## Risks / gotchas

- **created_by null:** with no auth, cart build sets `created_by` = the switched
  user (Task A). Fine, but not "auth".
- **Charge failures:** seeded PMs use `pm_card_visa` (always succeeds). Real
  Stripe key required for live charges; without it, seed writes placeholder PMs
  and checkout will mark `failed` — demo needs `STRIPE_SECRET_KEY` + `npm run seed`.
- **Realtime RLS:** anon delivery can be suppressed if the RLS read policy didn't
  apply; the migration handles it, but verify on the actual project.
- **Stale `.next`:** after adding routes/pages, if dev 500s with "client manifest",
  `pkill -f "next dev"; rm -rf web/.next`.
