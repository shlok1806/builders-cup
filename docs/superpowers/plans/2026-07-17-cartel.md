
# Cartel Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship Cartel — describe → agent builds cart → deterministic policy split → realtime approval → multi-card collection at checkout → dashboard — working end-to-end as a mobile-first installable PWA for a 3-minute live demo.

**Architecture:** Next.js App Router (mobile-first PWA UI + API routes as backend), Supabase (Postgres + Realtime), OpenAI (two structured-output calls only), Stripe sandbox (Payments — Customers + PaymentIntents, no Issuing). The LLM never touches the money path; splitting, thresholds, and charging are deterministic code. Demo access: laptop hosts the app on `0.0.0.0` while joined to the phone's hotspot/local LAN; the phone opens `http://<laptop-local-ip>:3000`, then installs the site as the PWA.

**Tech Stack:** Next.js 15 (App Router, TS), Tailwind, built-in web app manifest/PWA metadata, `@supabase/supabase-js`, `stripe`, `openai`, `vitest` (split-engine test only).

## Global Constraints

- **No auth.** Seeded users; a client-side dropdown sets "who am I"; the approval device uses `?user=<id>`.
- **Mobile-first PWA.** Optimize every screen for phone width first. Demo path is not native mobile: phone provides the hotspot/local LAN, laptop joins it and serves the app from its local IP, phone opens that local URL and adds it to the home screen as the PWA.
- **LLM in exactly two places:** cart builder (Call A) and policy compiler (Call B). Both use OpenAI **structured outputs** (`response_format: json_schema`), validate before use, and have a cached fallback keyed to the exact demo input.
- **Policy enum is exactly three:** `exclude_category | approval_threshold | split_weight`. `weekly_cap` and `pay_from` are cut.
- **Money is deterministic.** Split spec is §12 of `architecture.md` — treat it as the contract. Largest-remainder rounding; per-user cents sum exactly to each line total.
- **$0 users get no charge** (Stripe rejects `amount: 0`). Checkout guards on `status !== 'charged'` + idempotency key per (purchase, user, **amount**).
- **Realtime requires** `alter publication supabase_realtime add table approvals` and a browser subscription with the **anon** key — **anon delivery is a Phase-0 blocking gate** (enable RLS + a permissive read policy on `approvals` if RLS-off suppresses it). `GET /api/approvals` backs the poll fallback + device initial load.
- **Amounts are integer cents** everywhere. Currency `usd`.
- **All money-path logic is unit-tested once** against the canonical seed cart (§12). Other features get a smoke-verify, not a suite — this is a hackathon.
- **Feature freeze at hour 6.** Last 30 min: rehearse only, no new code.
- **Env vars:** `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `OPENAI_API_KEY`, `STRIPE_SECRET_KEY`.

- [ ] **Step 3: Implement** per §12 order (exclude → orphan-restore → weight → largest-remainder → flag):

```ts
import type { Member, Line, SplitResult, Allocation, Flag } from './types'

export function computeSplit(lines: Line[], members: Member[]): SplitResult {
  const allocations: Allocation[] = []
  const flagged: Flag[] = []
  for (const line of lines) {
    // 1-2. exclusions, with orphan restore
    let onLine = members.filter(m => !m.excludedCategories.includes(line.category))
    if (onLine.length === 0) onLine = members
    // 3. weights
    const totalWeight = onLine.reduce((s, m) => s + m.weight, 0)
    // 4. largest-remainder rounding
    const raw = onLine.map(m => (line.lineTotalCents * m.weight) / totalWeight)
    const floors = raw.map(Math.floor)
    let remainder = line.lineTotalCents - floors.reduce((s, n) => s + n, 0)
    const order = raw
      .map((r, i) => ({ i, frac: r - floors[i] }))
      .sort((a, b) => b.frac - a.frac)
    const cents = floors.slice()
    for (let k = 0; k < remainder; k++) cents[order[k].i]++
    onLine.forEach((m, i) =>
      allocations.push({ itemId: line.itemId, userId: m.userId, amountCents: cents[i] }))
    // 5. flag: first (lowest-index) surviving member whose threshold the line total hits
    const approver = onLine.find(
      m => m.approvalThresholdCents != null && line.lineTotalCents >= m.approvalThresholdCents)
    if (approver)
      flagged.push({ itemId: line.itemId, approverId: approver.userId,
        rule: `over $${(approver.approvalThresholdCents! / 100).toFixed(0)} approval threshold` })
  }
  return { allocations, flagged }
}
```

### Task P3.3b: Pending-approvals query (poll fallback + device initial load) — F5 support

**Files:** Create: `app/api/approvals/route.ts`

**Interfaces — Produces:** `GET /api/approvals?user=<id> → { pending:[{id,purchaseId,purchaseItemId,approverId,rule,itemName,amountCents}] }`
